docker镜像制作

发表于2022-12-15|更新于2022-12-15|云原生
|字数总计:4.5k|阅读时长:20分钟|阅读量:

镜像是容器的模板，那么镜像是怎么制作的呢？docker制作镜像有两种方式。第一种方式在一个基础容器上手动配置然后提交为镜像。第二种方式通过dockerfile文件自动制作镜像。我们一般用第二种方式制作镜像

手动制作镜像

这种方式制作流程如下：先选用一个基础容器，然后在该容器上安装基础命令、配置运行环境、安装软件和修改相关配置文件，最后再将该容器提交为一个新镜像即可。此种方式制作可以参考如下文章：通过yum制作nginx镜像:http://www.yunweipai.com/34818.html;通过编译nginx制作nginx镜像:http://www.yunweipai.com/34820.html

dockerfile制作

这种方式制作镜像相当于将上述手动制作镜像的过程写成脚本，不过采用的是dockerfile格式而不是常用的shell。

dockerfile是一个特殊格式的文本文件，它由一条条命令组成。Docker程序负责将每条命令翻译为对应的linux指令并根据指令生成新的镜像。后期镜像需要变更时，直接修改对应的dockerfile文件即可重新生成新的镜像。

dockerfile文件格式要求如下：

  • 每一行以Dockerfile的指令开头，指令不区分大小写，但是惯例使用大写
  • 使用 # 作为注释
  • 每一行只支持一条指令，每条指令可以携带多个参数
  • 指令按文件的顺序从上至下执行
  • 每条指令都是独立运行的，对下一条指令不会有任何影响。比如某条指令执行cd /usr/local/src，下一条指令初始时依旧是在工作目录。
  • 每个指令的执行会生成一个新的镜像层，为了减少分层和镜像大小，尽可能将多条指令合并成一条指令
  • 制作镜像一般可能需要反复多次，每次执行dockfile都按顺序执行，从头开始，已经执行过的指令会被缓存，不需要再执行。后续如果有一行指令没执行过，再往后的指令将会重新执行，所以为加速镜像制作，建议将最常变化的内容放在dockerfile文件的最后面

Dockerfile指令

FROM

FROM指令用于指定基础镜像,该指令必须放在有效指令的第一行。建议使用官方镜像作为基础镜像。该命令格式如下：

FROM [--platform=<platform>] <image>[:<tag>] [AS <name>]

说明：
    --platform 指定镜像的平台，比如：linux/amd64, linux/arm64, or windows/amd64
    tag 是可选项，如果不指定，默认为latest

LABEL

LABLE命令可以指定镜像元数据，比如镜像作者。该命令格式如下：

LABEL <key>=<value> <key>=<value> <key>=<value> ...

范例：

LABEL "com.example.vendor"="ACME Incorporated"
LABEL com.example.label-with-value="foo"
LABEL version="1.0"

# 多标签
LABEL multi.label1="value1" multi.label2="value2" other="value3"
LABEL multi.label1="value1" \
      multi.label2="value2" \
      other="value3"

可以通过docker inspect查看LABLE

RUN

RUN指令是用来调用SHELL命令的。一个文件中该指令可以有多个。每一个RUN指令运行后都会生成一层，所以尽可能将该指令合并成一条指令。该命令格式如下：

#shell 格式： 
RUN <命令> 

#exec 格式： 
RUN ["可执行文件", "参数1", "参数2"]

范例：

RUN echo '<h1>Hello, Docker!</h1>' > /usr/share/nginx/html/index.html
RUN ["/bin/bash", "-c", "echo hello world"]
RUN yum -y install epel-release \
      && yum -y install nginx \
      && rm -rf /usr/share/nginx/html/*
      && echo "<h1> docker test nginx </h1>" > /usr/share/nginx/html/index.html

COPY

该指令将复制本机的文件到容器中。该指令格式如下：

COPY <src>... <dest>
COPY ["<src1>",... "<目标路径>"]

说明：

  • 相对路径基于Dockerfile所在目录
  • 可以指定多个文件以及使用通配符
  • 使用该命令，源文件的各种元数据都会保留。比如读、写、执行权限、文件变更时间等。

范例：

COPY hom* /mydir/    
COPY hom?.txt /mydir/

ADD

该指令是COPY指令的升级版。可以复制宿主机以及URL文件到镜像中，如果是压缩包复制之后还会自动解压

ADD [--chown=<user>:<group>] <src>... <dest>
ADD [--chown=<user>:<group>] ["<src>",... "<dest>"]

说明：

  • 如果src是本地打包或压缩文件，如gzip, bzip2 ，xz ，将解包
  • 如果src是一个 URL ，下载后的文件权限自动设置为 600

范例：

ADD test relativeDir/          # adds "test" to WORKDIR/relativeDir/

ADD test /absoluteDir/         # adds "test" to /absoluteDir/

ADD --chown=55:mygroup files* /somedir/
ADD --chown=bin files* /somedir/
ADD --chown=1 files* /somedir/
ADD --chown=10:11 files* /somedir/

ADD ubuntu-xenial-core-cloudimg-amd64-root.tar.gz /

CMD

该指令用来指定容器启动时默认执行的命令。Dockerfile文件中只能有一条CMD指令，如果指定了多条则只有最后一条被执行。如果Dockerfile中包含ENTRYPOINT，那么CMD就是给ENTRYPOINT提供参数的。如果容器启动过程中指定了运行的指令，该指令也会覆盖掉Dockerfile文件中的CMD命令。

即：如果docker run没有指定任何的执行命令或者dockerfile里面也没有ENTRYPOINT，那么就会使用CMD指定的命令

# 使用 exec 执行，推荐方式，第一个参数必须是命令的全路径
CMD ["executable","param1","param2"] 

# 在 /bin/sh 中执行，提供给需要交互的应用；
CMD command param1 param2 

# 提供给 ENTRYPOINT 的默认参数；
CMD ["param1","param2"]

范例：

CMD ["nginx", "-g", "daemon off;"]

ENTRYPOINT

功能类似于CMD，配置容器启动后执行的命令及参数，并且不可被 docker run 提供的参数覆盖，而是追加。使用CMD要在运行时重新写命令才能追加运行参数，ENTRYPOINT则可以在运行时接受新参数

如果docker run命令有参数，那么参数全部都会作为ENTRYPOINT的参数。如果docker run后面没有额外参数，但是dockerfile中的CMD里有（即上面CMD的第三种用法），那么CMD的全部内容会作为ENTRYPOINT的参数

可以通过docker run –entrypoint 参数 运行时替换

每个 Dockerfile 中只能有一个 ENTRYPOINT，当指定多个时，只有最后一个生效。

# 使用 exec 执行
ENTRYPOINT ["executable", "param1", "param2"]

# shell中执行
ENTRYPOINT command param1 param2

范例：

FROM ubuntu:18.04
RUN apt update \
&& apt -y install  curl \
&& rm -rf /var/lib/apt/lists/*
ENTRYPOINT [ "curl", "-s","https://ip.cn"]

ENV

指定一个环境变量，会被后续 RUN 指令使用，并在容器运行时保持

ENV <key> <value>
ENV <key1>=<value1> <key2>=<value2>...

范例：

ENV VERSION=1.0 DEBUG=on NAME="Happy Feet"
ENV PG_MAJOR 9.3
ENV PG_VERSION 9.3.4
RUN curl -SL http://example.com/postgres-$PG_VERSION.tar.xz | tar -xJC /usr/src/postgress && …
ENV PATH /usr/local/postgres-$PG_MAJOR/bin:$PATH

ARG

该指令也是指定环境变量。和ENV不同的是，容器运行时不会存在这些环境变量。如果和ENV同名，也会被ENV覆盖。可以用 docker build –build-arg <参数名>=<值> 来覆盖。

ARG <name>[=<default value>]

VOLUME

Docker容器启动时会创建一个可读写“层”，该层会随着容器的删除而删除。VOLUME指令会将宿主机上的目录/var/lib/containers/storage/volumes/<id>/_data 挂载至VOLUMN指令指定的容器目录。即使后期容器删除，此宿主机的目录仍会保留，从而实现容器数据的持久化保存。Docker-20.10.21位置在/var/lib/docker/volumes/<id>/_data 

VOLUME <路径>
VOLUME ["<容器内路径1>", "<容器内路径2>"...]

范例：

# 制作Dockerfile
[root@192 alpine]# cat Dockerfile

LABEL autor="qiyankuo"
VOLUME ["/testdata","/testdata2"]

EXPOSE

声明Docker容器需要暴露的端口号。该指令仅仅声明容器打算使用什么端口而已，并不会自动在宿主机上进行端口映射。因此，在启动容器时需要通过-P或-p ，Docker主机才会分配一个端口转发到指定容器暴露的端口。

EXPOSE <端口1> [<端口2>...]

WORKDIR

该指令为后续的 RUN、CMD、ENTRYPOINT 指令配置工作目录。当容器运行后，进入容器内的默认目录

WORKDIR /path/to/workdir

示例：

#两次run不在一个环境内，可以使用WORKDIR
WORKDIR  /app
RUN echo "hello" > hello.txt
RUN echo "world" > world.txt

#可以使用多个 WORKDIR 指令，后续命令如果参数是相对路径，则会基于之前命令指定的路径。例如
WORKDIR /a
WORKDIR b
WORKDIR c
RUN pwd
则最终路径为 /a/b/c

ONBUILD

该指令用来配置当创建当前镜像的子镜像时，会自动触发执行的指令

ONBUILD [INSTRUCTION]

示例：

Dockerfile 使用如下的内容创建了镜像 image-A。
...
ONBUILD ADD . /app/src
ONBUILD RUN /usr/local/bin/python-build --dir /app/src...

如果基于image-A创建新的镜像时，新的Dockerfile中使用FROM image-A指定基础镜像时，会自动执行ONBUILD指令内容，等价于在后面添加了两条指令。

FROM image-A

#Automatically run the following
ADD . /app/src
RUN /usr/local/bin/python-build --dir /app/src

使用ONBUILD指令的镜像，推荐在标签中注明，例如ruby:1.9-onbuild

USER

指定运行容器时的用户名或 UID，后续的RUN等指令也会使用指定的用户身份

USER <user>[:<group>] 
或
USER <UID>[:<GID>]

当容器中运行的服务不需要管理员权限时，可以先建立一个特定的用户和用户组，为它分配必要的权限，使用 USER 切换到这个用户

重点

  • 使用USER指定用户时，可以使用用户名、UID 或 GID，或是两者的组合
  • 使用USER指定用户后，Dockerfile中后续的命令 RUN、CMD、ENTRYPOINT都将使用该用户

运行时可以使用 docker run -u 替换默认设置的用户

HEALTHCHECK

检查容器的健康性：

HEALTHCHECK [选项] CMD <命令> #设置检查容器健康状况的命令
HEALTHCHECK NONE #如果基础镜像有健康检查指令，使用这行可以屏蔽掉其健康检查指令

HEALTHCHECK 支持下列选项：
--interval=<间隔> ：两次健康检查的间隔，默认为 30 秒；
--timeout=<时长> ：健康检查命令运行超时时间，如果超过这个时间，本次健康检查就被视为失败，默认 30 秒；
--retries=<次数> ：当连续失败指定次数后，则将容器状态视为 unhealthy ，默认 3次。

STOPSIGNAL

该指令设置将被发送给容器的进行退出的系统调用信号。该信号可以是与内核syscall表中的位置匹配的有效无符号数字（例如9），也可以是SIGNAME格式的信号名称（例如SIGKILL）

STOPSIGNAL signal

.dockerignore

与.gitignore文件类似，生成构建上下文时Docker客户端应忽略的文件和文件夹。具体内容可以参照如下链接：https://blog.csdn.net/qianghaohao/article/details/87902806

制作nginx镜像

创建工作目录

[root@192 1.16]# pwd
/data/dockerfile/web/nginx/1.16

下载nginx源码包

[root@1.16]# wget  http://nginx.org/download/nginx-1.16.1.tar.gz

制作静态页面

[root@1.16]# mkdir app/
[root@1.16]# #echo  "Test Page in app" > app/index.html
[root@1.16]# #tar zcf app.tar.gz app

创建Dockerfile

[root@192 1.16]# cat Dockerfile 
FROM centos-base:v1
LABEL autor="qiyankuo"
ADD  nginx-1.16.1.tar.gz /usr/local/src
RUN cd /usr/local/src/nginx-1.16.1 \
    && ./configure --prefix=/apps/nginx \
    && make && make install \
    && rm -rf /usr/local/src/nginx* \
    && useradd -r nginx
ADD app.tar.gz /apps/nginx/html/

EXPOSE 80
CMD /apps/nginx/sbin/nginx && tail -f /etc/hosts

制作镜像

[root@192 1.16]# cat build.sh 
#!/bin/bash
#
docker build -t nginx-centos7:1.6.1

[root@192 1.16]# ./build.sh
